AMR, ECSA, and FFG Project Work

Two related but separate bodies of work: ECSA is a technical AI-for-science architecture for probabilistic AMR modeling; AMR-HORIZONT is a participatory governance and FFG Impact Innovation project package.

ECSA technical architecture

  • Dynamic graph/state-space system for AMR.
  • Resistance, uncertainty, and stability parameters.
  • Bayesian inference, GNN link prediction, optional RL.
  • Lab-in-the-loop feedback and gold-data generation.
  • Regulatory by design: MDR, ISO, AI Act, GDPR, FAIR.

AMR-HORIZONT project package

  • 12-month FFG Impact Innovation Social Track plan.
  • Budget and work-package logic.
  • Stakeholder and LOI strategy.
  • Q&A and red-flag preparation.
  • Scenario-based AP5 testing with no patient data.
Boundary: ECSA is not presented as clinical validation. AMR-HORIZONT is not presented as funded unless separate confirmation is provided.

Clinical

Empirical antibiotic therapy under sepsis suspicion, balancing patient benefit, time pressure, resistance management, local data, and transparency.

Two AMR tracks, one capability pattern

The AMR work has two tracks that should not be merged into one claim. ECSA is a technical AI architecture for probabilistic AMR modeling. AMR-HORIZONT is a participatory One-Health governance/project package prepared for FFG Impact Innovation. Together they show both algorithmic thinking and project execution, but each has a different evidence type and boundary.

ECSA evidence type

Architecture proposal, AMR dossier, PCT/utility-model-style technical drafting, risk and regulatory sections, lab-in-the-loop methodology, model layer with Bayesian/GNN/RL components.

AMR-HORIZONT evidence type

12-month project plan, budget logic, stakeholder map, methodology, Theory of Change, LOI/support templates, evaluator Q&A, red-flag handling, AP5 scenario testing, output decision map.

Project-management signal

For employers or collaborators, AMR-HORIZONT is useful because it shows that I can organize a complex project beyond code: problem framing, stakeholder segmentation, budget, external expertise, evidence-limited answers, risk mitigation, deliverables, and scenario-based validation. It should be described as application and planning evidence, not as proof of funded execution unless a separate funding confirmation is provided.

Why ECSA belongs in the Fellowship package

ECSA should not lead the Fellowship application, because the Secure AI proof is stronger in the agent-infrastructure work. It belongs as the future project answer: a concrete AI-for-science direction where probabilistic modeling, uncertainty, lab feedback, governance, and model accountability all matter.

The proposed system models antimicrobial resistance as a dynamic graph/state-space problem. Instead of returning one black-box prediction, it carries resistance, uncertainty, and stability through the model, updates those parameters with new evidence, and returns risk/trajectory information together with the limits of the current model.

How it connects back to the AI systems

The personal agent work and ECSA are not random separate interests. The same engineering concerns transfer: model routing, evaluator loops, memory provenance, uncertainty, auditability, and human-in-the-loop control. Miguel-like infrastructure could support literature ingestion, hypothesis generation, experiment tracking, model evaluation, and governance documentation, while ECSA gives that infrastructure a serious scientific target.
